## C++ Micro-Benchmarks

Standalone C++ benchmarks measuring the core data structure changes.
500,000 items inserted, 5,000,000 lookup operations.

| Operation | `std::map` (baseline) | `std::unordered_map` (perf) | **Speedup** |
|-----------|----------------------|----------------------------|-------------|
| Insert (500K items) | 335 ms | 102 ms | **69.7% faster** |
| Lookup (5M ops) | 2,700 ms | 285 ms | **89.5% faster** |
| Erase (250K items) | 408 ms | 92 ms | **77.4% faster** |
| Iterate (500K items) | 363 ms | 112 ms | **69.2% faster** |
| Vector push_back (no reserve) | 941 us | â€” | baseline |
| Vector push_back (with reserve) | â€” | 646 us | **31.4% faster** |

### Why it matters

Map block lookups by `v3s16` position are one of the hottest code paths in the engine.
During active gameplay with multiple players, millions of these lookups happen per second
for block validation, lighting, meshing, and network sync. The O(1) average-case lookup
of `unordered_map` vs O(log n) for `std::map` translates directly to higher tick rates
and reduced server lag under load.

### Analysis

The Lua-level benchmark shows smaller differences because the Luaâ†’C++ API call
overhead dominates individual operation timing. The VoxelManip stages (B) show
regression due to the larger memory footprint of `unordered_map` node tables
affecting cache locality during bulk sequential scans. However, the ABM-like
random-access pattern (D2) shows the expected improvement.

The real-world gains will be most visible during active multiplayer gameplay
with many players generating and modifying blocks simultaneously, where random
map block lookups dominate.
